\subsection{Overview}
The thesis uses an empirical, experimental methodology:
\begin{enumerate}
    \item Implement an \gls{llm}-driven control loop for a simulated heterogeneous warehouse environment.
    \item Define the main comparison axes and evaluation metrics.
    \item Run controlled experiments (fixed environment ids and seeds) across multiple \gls{llm} models, planning architectures, and prompt variants.
    \item Analyze throughput, safety, and coordination outcomes as well as qualitative failure modes.
\end{enumerate}

\subsection{Environment: Battery-TA-RWARE}
Battery-TA-RWARE is a discrete-time, grid-based warehouse simulation derived from \gls{tarware}~\cite{krnjaic2022scalable}.
At any time, a set of shelves is requested. Carrier \glspl{agv} must transport requested shelves to goal locations and then return them to storage.
Picker agents are required to meet \glspl{agv} at shelf locations for load and unload interactions, creating explicit heterogeneous coordination constraints.
Agents additionally have a battery state that decreases with movement and interactions; charging stations are available at fixed locations.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.95\textwidth]{figures/tarware_explanation.png}
    \caption{Illustration of the task-assignment warehouse setting used in this thesis. Two heterogeneous roles must coordinate: carrier \glspl{agv} transport shelves, while picker agents support loading and unloading at shelf locations. Battery dynamics add a safety constraint that can force agents to recharge.}
    \label{fig:tarware_overview}
\end{figure}

The environment exposes a discrete action space where each action corresponds to selecting a target location:
\begin{itemize}
    \item \textbf{Shelf actions:} choose a shelf location (e.g., to pick up or return a shelf).
    \item \textbf{Goal actions:} choose a delivery station (primarily relevant when an \gls{agv} carries a requested shelf).
    \item \textbf{Charging actions:} choose a charging station to recover battery.
\end{itemize}
Low-level motion toward the selected target is handled internally by the environment (A* path planning and collision resolution), while the controller is evaluated on the quality of these high-level decisions.

\subsection{Language-Model Control Loop}
\label{sec:llm_loop}
The \gls{llm} policy is implemented as a text-in / discrete-action-out controller:
\begin{enumerate}
    \item \textbf{State summarization:} extract salient features of the global state (requested shelves and their action ids, per-agent position/target/battery/carrying status, charging occupancy).
    \item \textbf{Candidate action set:} compute a valid-action mask from the environment; optionally restrict the candidate list shown to the \gls{llm} to reduce prompt size.
    \item \textbf{Prompting:} generate a per-agent prompt describing the agent role and the current state, including explicit mappings from requested objects to action ids.
    \item \textbf{Querying:} send the prompt to a local inference server (Ollama) and obtain a text response.
    \item \textbf{Parsing:} parse an integer action id (and optionally an action hold duration) from the response.
    \item \textbf{Validation:} validate the parsed action id against the current valid-action mask; invalid outputs are discarded and trigger replanning.
    \item \textbf{Execution:} apply the validated joint action in the environment for one step (or hold the action for multiple steps, depending on configuration).
\end{enumerate}

The controller is best described as a discrete-time, step-synchronous supervisory loop with conditional replanning.
The environment advances one simulation step at a time, and at each step the controller reevaluates agent state and action validity.
However, this does not imply that every agent is queried at every step.
Instead, the experiments use two related control schemes.

\begin{itemize}
    \item \textbf{Step-synchronous replanning with environment-owned continuation:} agents that are currently available for planning may receive a new high-level action, while busy agents are not replanned and continue their current task until the environment releases them.
    \item \textbf{Step-synchronous replanning with action persistence and conditional override:} accepted actions may remain active for multiple future steps, so agents are not re-queried while the persisted action remains valid. Replanning is triggered when a persisted action expires, becomes invalid, or when the controller decides intervention is needed, including selected busy-agent cases.
\end{itemize}

This design sits between two extremes.
A fully scheduled scheme that queries every agent at every step would generate many unnecessary \gls{llm} calls, while a purely event-triggered scheme is less natural in this setting because useful triggers such as blocked progress, invalid persisted actions, expiring commitments, or changing support needs are only known after the system state is checked.
The adopted design therefore combines regular per-step observability with selective replanning.
This is also consistent with practical supervisory control in warehouse systems, where the fleet is typically checked at regular intervals, ongoing tasks continue by default, and new task-level decisions are issued only when the current situation justifies intervention.

Two design choices are central for feasibility:
\begin{itemize}
    \item \textbf{Grounding via action ids:} the \gls{llm} does not output coordinates or free-form text commands; instead it selects from discrete action ids that are already defined by the environment.
    \item \textbf{Hard validation:} action masks enforce constraints such as invalid goals for picker agents and busy-state restrictions. Invalid \gls{llm} outputs are never executed.
\end{itemize}

\subsection{Experiment Families and Objectives}
The implemented experiments are grouped into three objective families.

\paragraph{Objective 1: maximize total shelf deliveries.}
The first family focuses on raw throughput. These experiments are stored under \nolinkurl{results/shelf} and were designed to answer two questions: whether a centralized or shared-context planning architecture is more effective, and whether natural-language or \gls{json}-style prompting is the more suitable representation for that objective. The family was implemented in two architectures:
\begin{itemize}
    \item \textbf{Centralized planner:} \nolinkurl{scripts/run_centralized_llm_experiments.py}. One \gls{llm} prompt receives the global state and returns a joint plan for all non-busy agents.
    \item \textbf{Shared-context decentralized planner:} \nolinkurl{scripts/run_shared_context_llm_expt.py}. Each agent is planned separately, but each prompt still contains a shared global summary of requests, charging occupancy, and waiting support relations.
\end{itemize}
For both architectures, two prompt styles were evaluated:
\begin{itemize}
    \item \textbf{Natural-language form:} the prompt and response use a compact textual format.
    \item \textbf{\gls{json} form:} the prompt and response use a more structured object-and-array representation.
\end{itemize}

\paragraph{Objective 2: optimize \gls{llm} calls while maintaining delivery throughput.}
The second family is implemented in \nolinkurl{scripts/run_obj2_shared_context_llm.py} and stored under \nolinkurl{results/calls}. This family keeps the shared-context, per-agent architecture, but changes the objective from delivery-only to a joint objective: increase shelf deliveries while reducing unnecessary \gls{llm} queries. In practice, the runner uses action holding and conditional replanning so that the model is not queried for every agent at every environment step. This family was used primarily to study the practical trade-off between throughput, model size, and call efficiency, and more broadly to test whether the \gls{llm}-based control framework remained useful when the objective changed.

\paragraph{Objective 3: battery-aware objective.}
A third family was planned for explicitly battery-oriented evaluation and corresponding result storage under \nolinkurl{results/battery}. At the time of writing, this experimental branch had not yet produced completed result files, so it is described as planned work rather than evaluated work.

\subsection{Planning Architectures}
The experiments compare two distinct ways of integrating the \gls{llm} into the decision loop.

\paragraph{Centralized planning.}
In the centralized architecture, one \gls{llm} call plans all agents that currently need a decision. This design exposes the full global state and all relevant candidate actions in one prompt. Its intended advantage is explicit joint coordination. Its main risk is prompt overload: as the number of agents and candidate actions increases, the prompt becomes large and more difficult to parse reliably.

\paragraph{Shared-context per-agent planning.}
In the shared-context architecture, the \gls{llm} plans one agent at a time. Each prompt still includes global information, but the final decision concerns only the current agent. This reduces the output burden and simplifies parsing, at the cost of requiring coordination to emerge from repeated local decisions rather than from one explicit joint plan. In the thesis evaluation, this architecture is the main vehicle for comparing prompt forms and model sizes under a common decision structure.

The \nolinkurl{run_obj2_shared_context_llm.py} variant extends the shared-context architecture with explicit action-hold durations and logging for call-efficiency analysis. It therefore remains decentralized in prompt structure while adding stronger runner-side control over when replanning is allowed.

\subsection{Prompt Structures}
The prompt comparison in this thesis is not limited to natural language versus \gls{json}. The prompt design varies along two axes:
\begin{itemize}
    \item \textbf{Planning scope:} centralized joint planning versus per-agent planning with shared context.
    \item \textbf{Output structure:} natural-language responses versus \gls{json} responses.
\end{itemize}

Across the different scripts, the prompts evolved to include:
\begin{itemize}
    \item explicit requested-shelf to action-id mappings,
    \item role-specific rules for \glspl{agv} and pickers,
    \item support relations such as waiting-for-picker context,
    \item battery-state abstractions,
    \item candidate action filtering or prioritization,
    \item and, in Objective~2, an explicit \nolinkurl{Steps} field intended to reduce unnecessary \gls{llm} calls.
\end{itemize}

The purpose of this progression was experimental rather than aesthetic: to determine which combination of planning scope and prompt structure yields valid, productive macro actions under the constraints of the warehouse task. In particular, the prompt comparison was intended to separate two questions: whether planning should be centralized or per-agent, and whether the same decision content is handled better in natural-language form or in a more structured \gls{json} form.

\subsection{Comparison Axes}
The current report focuses on internal comparisons within the \gls{llm}-based control framework rather than on comparison against an external baseline. The main comparison axes are:
\begin{itemize}
    \item model size,
    \item planning architecture (centralized versus shared-context per-agent),
    \item prompt form (natural language versus \gls{json}),
    \item and objective formulation (shelf-delivery only, call-aware shelf delivery, and planned battery-aware control).
\end{itemize}

\subsection{Evaluation Metrics}
The main metric is delivery throughput, expressed as a pick rate (deliveries per hour) derived from the number of delivered shelves and the simulated episode duration.
Additional metrics are collected to capture coordination and safety:
\begin{itemize}
    \item total shelf deliveries,
    \item number of clashes and deadlocks/stuck events,
    \item distance travelled by each agent type,
    \item episode return (environment reward),
    \item episode length and inference speed/latency (when measured).
\end{itemize}

\subsection{Experimental Protocol (Current Stage)}
To support reproducibility during prompt iteration, the reported experiment sessions were run with fixed environment ids and a fixed seed.
This choice makes behavioural debugging easier, but it also limits statistical confidence. The current thesis therefore treats the results as experimental evidence about architecture and prompt behaviour, not as final performance claims.

The progression of the experimental campaign was:
\begin{enumerate}
    \item Objective~1 shelf-delivery experiments, comparing centralized and shared-context planning and comparing natural-language and \gls{json} prompt forms,
    \item Objective~2 call-aware shelf-delivery experiments, using a later shared-context controller with stronger prompt structure and safeguards,
    \item Objective~3 battery-aware experiments, planned but not yet completed.
\end{enumerate}

One broader aim of this progression was to test whether the same general \gls{llm}-based control framework could adapt as the stated objective changed. Section~\ref{sec:result_discussion} therefore first analyzes Objective~1 as the shelf-delivery baseline, then Objective~2 as a changed objective using a later and more structured shared-context controller, and finally notes Objective~3 as planned future evaluation. In the current simulation setup, a run can achieve at most 20 deliveries, and the results chapter uses that ceiling to contextualize reported scores.

\subsection{Delimitations}
To keep the study focused and interpretable at the current stage, the following delimitations apply:
\begin{itemize}
    \item The work is evaluated in a simulated warehouse environment rather than on physical robots.
    \item The controller is responsible only for high-level target selection; low-level navigation and collision handling are performed by the environment.
    \item The evaluated language models are prompted general-purpose models accessed through a local inference server and are not fine-tuned for this task.
    \item The reported experiments use fixed seeds and a limited number of completed sessions, so the results should be interpreted as exploratory rather than statistically conclusive.
    \item The battery-aware objective family had not yet produced completed aggregate results at the time of writing.
\end{itemize}

\subsection{Ethical and Sustainability Considerations}
Although the present work is simulation-based, it raises practical issues that remain relevant for real multi-robot deployments.
The primary ethical concern is safety and reliability: generated decisions may be infeasible, unstable, or unproductive if they are not constrained by system-side checks.
For that reason, the studied control loop does not execute free-form outputs directly. It uses grounded action identifiers, candidate shaping, explicit feasibility validation, and fallback behavior so that unsafe or clearly invalid actions are filtered before execution.
The logging of prompts, parsed outputs, executed actions, and observed step consequences also improves traceability, which is important when analyzing failures such as long no-delivery tails, stale waiting loops, or battery-related breakdowns.

From a sustainability perspective, the main trade-off is between model capability and computational cost.
Larger language models generally require more inference time and compute, while smaller models may require stronger prompt structure and validation to remain effective.
This thesis therefore treats call efficiency, unnecessary replanning, and coordination overhead as practically relevant concerns rather than only implementation details.
Even with safeguards, the current study does not eliminate all risks: fixed-seed evaluation, limited scenario coverage, and the absence of physical deployment mean that conclusions about safety, efficiency, and generalization remain provisional.
